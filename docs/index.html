>_TRADERVIEW — DESKTOP & WEB FROM ONE WORKSPACE

Two binaries, one crate graph. A Tauri v2 desktop app that downloads and runs an embedded PostgreSQL on first launch and auto-logs in a local user — and an axum web server that talks to an external Postgres with argon2 + JWT auth for multi-user deployment. Both binaries depend on the same traderview-{core,db,import} library crates; the vanilla JS + uPlot frontend is shared verbatim. Executions are the atom; trades are FIFO-derived; the journal is markdown.

Quickstart

Clone the repo, vendor uPlot, then pick a deploy target:

# 1. clone + vendor uPlot
git clone https://github.com/MenkeTechnologies/traderview
cd traderview
./scripts/vendor-uplot.sh

# 2a. desktop — Tauri v2, embedded Postgres
cargo tauri dev          # first launch downloads PostgreSQL (~80 MB)

# 2b. web — axum, external Postgres
docker compose up -d postgres
export DATABASE_URL=postgres://traderview:traderview@localhost:5432/traderview
export TRADERVIEW_JWT_SECRET=$(openssl rand -hex 32)
cargo run -p traderview-web --bin server
# open http://localhost:8080

The desktop app is a single user, fully offline. The web binary is multi-user with registration and JWT bearer auth. Same schema, same migrations, same FIFO roll-up, same frontend, same API surface.

Architecture — One Workspace, Two Deploy Targets

┌──────────────────────────────────────┐ │ frontend/ (vanilla JS + uPlot) │ │ dashboard / trades / journal / ... │ └──────────────┬───────────────────────┘ │ HTTP /api/* ┌────────────────┴────────────────┐ │ │ ┌─────────▼──────────────┐ ┌────────────▼─────────────┐ │ src-tauri │ │ traderview-web (axum) │ │ (traderview-desktop) │ │ bin: server │ │ embedded postgres │ │ external postgres │ │ auto-login local user │ │ argon2 + JWT auth │ └─────────┬──────────────┘ └────────────┬─────────────┘ │ │ └────────────────┬────────────────┘ │ ┌────────────────▼─────────────────┐ │ traderview-{core, db, import} │ │ shared library crates │ └──────────────────────────────────┘

The desktop and web binaries are thin shells. All domain logic, all SQL, all broker parsing lives in the three library crates. The decision "embedded vs external Postgres" and "local auto-user vs multi-user with auth" is the only thing that distinguishes them.

Desktop · traderview-desktop

Tauri v2 shell. postgresql_embedded downloads a portable PostgreSQL on first launch (cached in ~/.theseus), data lives under $APP_DATA_DIR/traderview/pg/. Auto-creates a single local user and auto-logs in. Axum router runs on a random localhost port; the WebView talks to it via fetch. Postgres shuts down cleanly on window close.

Web · server

Axum 0.7 router on 0.0.0.0:8080 (configurable). Connects to external Postgres via DATABASE_URL. argon2 password hashing on registration, JWT bearer auth on every protected route. CORS allowlist via TRADERVIEW_CORS_ORIGIN. Refuses to start without TRADERVIEW_JWT_SECRET.

Shared · traderview-{core,db,import}

Three library crates that both binaries link. core owns models + FIFO roll-up + stats. db owns sqlx pool + migrations + embedded-Postgres lifecycle. import owns broker parsers (Webull first). One-way dependency graph: desktop and web both depend on all three; import and db both depend on core; nothing depends on desktop or web.

Schema — One Migration, 9 Tables

Single migration (migrations/0001_initial.sql, 149 LOC) defines the entire schema. Money is NUMERIC(20, 8) everywhere — no floats. Sides are typed enums.

TablePurpose
usersReal users (web) or one auto-created local user (desktop). Email and password_hash are nullable to accommodate the desktop auto-user. Unique on lower(email) when non-null.
accountsBroker accounts (webull, ibkr, tos, …). One user → many accounts. base_currency defaults to USD.
executionsOne row per fill — the atom. Unique on (account_id, broker_order_id, executed_at, symbol, side, qty, price) when broker_order_id is non-null, so re-importing the same statement is idempotent.
tradesFIFO-derived from executions. Materialized for fast UI queries. Carries status (open / closed), side (long / short), entry/exit averages, gross P&L, fees.
trade_executionsMany-to-many: which executions composed each trade. Lets you click a trade and see every fill that built the position.
trade_tagsFree-form tags per trade — "scalp", "earnings", "momo", whatever the journal needs.
journal_entriesMarkdown notes, either keyed to one trade (trade_id) or to a calendar day (day).
importsAudit trail — every CSV upload, the broker, the row count, the file hash. Lets you re-parse the same source if the importer improves.
_sqlx_migrationssqlx's own migration tracker.

Enum types: side_t = (buy, sell, short, cover) on executions; trade_side_t = (long, short) and trade_status_t = (open, closed) on trades.

HTTP API — 10 Routes Under /api/

MethodPathPurpose
GET /health Liveness probe. No auth.
GET /config Server config + auth mode (local for desktop, multi for web). Frontend reads this on boot to decide whether to show the login screen.
POST /auth/register Create user. Web mode only — desktop refuses.
POST /auth/login Exchange (email, password) for a JWT bearer token. argon2 verify.
GET /auth/me Current user from bearer token.
GET /accounts List broker accounts for the authenticated user.
GET /trades List trades. Filter by symbol, status, opened_at range.
GET /stats/summary Summary stats — total trades, win rate, average win/loss, expectancy, profit factor.
GET /stats/equity Equity curve points for uPlot. [timestamp, cumulative_pnl] rows.
GET /journal/:day Markdown journal entries for a calendar day.

Desktop mode auto-logs in as the local user; the frontend talks to the embedded server on a random localhost port. Web mode requires POST /api/auth/login → returns JWT → sent as Authorization: Bearer … on subsequent calls.

Frontend — Vanilla JS + uPlot, Five Views

Zero npm. Zero bundler. Zero JS framework. 384 lines of JS, 259 lines of CSS, no build step. The entire frontend ships as static files under frontend/ and is served by axum's tower-http fs middleware in web mode or loaded directly from disk by Tauri in desktop mode.

Dashboard

js/app.js. Summary stats card (win rate / expectancy / profit factor / total P&L) + equity curve drawn by js/charts.js using uPlot.

Trades

js/trades.js. Filterable, sortable trade table. Filter by symbol, status, date range, tag. Click a trade to drill into its component executions.

Journal

js/journal.js. Per-day and per-trade markdown notes. Saved server-side in journal_entries; markdown is rendered client-side.

Import

js/import.js. Broker CSV upload, dedupe report, re-parse from imports table.

Accounts

js/auth.js. List broker accounts, add / edit / remove. Auth screens (login / register) for web mode.

js/api.js wraps fetch — it adds the JWT bearer header in web mode and a no-op header in desktop mode. uPlot is vendored under frontend/lib/ by ./scripts/vendor-uplot.sh — pinned, reproducible, no CDN at runtime.

Importing — Webull First, Mapping Wizard Later

Export Webull Account Statement → Orders as CSV. Drop the file into the Import view. The parser:

  1. Inserts the raw row into imports for audit trail.
  2. Maps each row to an execution and inserts under the dedupe key (account_id, broker_order_id, executed_at, symbol, side, qty, price).
  3. Re-runs the FIFO roll-up for affected (account_id, symbol) pairs and updates trades.

Re-importing the same CSV is idempotent — the unique constraint silently drops duplicate fills. Importing a new statement that overlaps an old one is also safe; only the new fills are inserted.

Other brokers (IBKR, TOS, Schwab, Fidelity) are scheduled. The importer ships a generic mapping-wizard backend once the Webull baseline is solid — you point at a CSV, the wizard maps columns to execution fields, and the mapping is saved per-broker per-user.

Why Not TraderVue?

TraderVue is hosted-only, subscription-priced, and your trades live on their server. traderview is single-binary desktop or self-hosted multi-user; your data never leaves your machine in desktop mode and never leaves your infrastructure in web mode. Same UI shape — dashboard, trades, journal, equity curve — but you own the database and the deploy.
